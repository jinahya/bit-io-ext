== Bugs

=== `ByteInputAdapter.align(int)` / `ByteOutputAdapter.align(int)` skips extra bytes when already aligned

When `bytes > 1` and `count % bytes == 0` (already at a multi-byte boundary), the loop calculates
`i = bytes - 0 = bytes` and reads/writes an unnecessary full alignment unit.

.ByteInputAdapter.java:82
[source,java]
----
for (int i = (bytes - (int) (this.count % bytes)); i > 0L; i--) {
----

.ByteOutputAdapter.java:79
[source,java]
----
for (int i = (bytes - (int) (this.count % bytes)); i > 0; i--) {
----

The condition should guard against the `remainder == 0` case:

[source,java]
----
final int remainder = (int) (this.count % bytes);
if (remainder > 0) {
    for (int i = bytes - remainder; i > 0; i--) {
        skip(Byte.SIZE);
        bits += Byte.SIZE;
    }
}
----

== Enhancements

=== `CountReader` / `CountWriter`: IOException silently wrapped in RuntimeException

`CountReader.newCountReader` and `CountWriter.newCountWriter` return `ToIntFunction` / `ObjIntConsumer`
which cannot declare checked exceptions.
IOException is caught and re-thrown as `RuntimeException`, making I/O failures invisible to callers
who expect checked exceptions.

.CountReader.java:44-50
[source,java]
----
return i -> {
    try {
        return i.readInt(true, size);
    } catch (final IOException ioe) {
        throw new RuntimeException("failed to read count from " + i, ioe);
    }
};
----

Consider defining custom functional interfaces that declare `throws IOException`.

=== `CountReader` / `CountWriter`: reflection-based `setCountReader`/`setCountWriter` default methods

The default `setCountReader` / `setCountWriter` implementations use reflection with a hardcoded field name
`"countReader"` / `"countWriter"`.
This bypasses compile-time checking and breaks if the field is renamed.

.CountReader.java:70-81
[source,java]
----
default void setCountReader(final ToIntFunction<? super BitInput> countReader) {
    ...
    final Field field = getClass().getDeclaredField("countReader");
    ...
}
----

Both `ByteArrayReader` and `ListReader` already override with direct field assignment,
making the reflective default unused.
Consider making the default method `abstract` or removing the reflective fallback.

=== `ListReader` / `ByteArrayReader`: no upper bound on count values

`countReader.applyAsInt(input)` can return `Integer.MAX_VALUE`, causing `new ArrayList<>(size)` or
`new byte[length]` to attempt massive allocations from untrusted input.

.ListReader.java:52-55
[source,java]
----
final int size = countReader.applyAsInt(input);
final List<E> value = new ArrayList<>(size);
----

.ByteArrayReader.java:191
[source,java]
----
final int length = countReader.applyAsInt(input);
final byte[] value = new byte[length];
----

Consider validating that the count is non-negative and optionally capping it with a configurable maximum.

=== `ByteArrayWriter.CompressedUtf8` / `ByteArrayReader.CompressedUtf8`: no bounds check on multi-byte sequences

The `++i` increments inside the UTF-8 switch cases can cause `ArrayIndexOutOfBoundsException`
if the byte array contains a truncated multi-byte sequence.

.ByteArrayWriter.java:172 (and similar in ByteArrayReader.java:147)
[source,java]
----
output.writeInt(true, 6, value[++i] & 0xFF);
----

Consider validating remaining array length before incrementing.

=== `Leb128Writer.OfUnsigned`: write format inconsistent with `Leb128Writer.OfSigned`

The unsigned writer emits continuation and data as separate 1-bit + 7-bit writes,
while the signed writer emits a single 8-bit byte.
Both produce identical bit streams, but the inconsistency hinders readability.

.Leb128Writer.java:83-84 (unsigned)
[source,java]
----
output.writeInt(true, 1, last ? 0 : 1);
output.writeInt(true, 7, group);
----

.Leb128Writer.java:133,136 (signed)
[source,java]
----
output.writeInt(true, Byte.SIZE, group);
output.writeInt(true, Byte.SIZE, (group | 0x80));
----

Consider unifying to a single 8-bit write style for consistency.

=== `Leb128Reader.OfUnsigned` / `Leb128Writer.OfUnsigned`: no overflow guard for values exceeding 63 bits

The unsigned LEB128 reader accumulates bits with `shift += 7` but never checks that `shift` stays
within `Long.SIZE`.
Malformed input with more than 10 continuation bytes would silently produce a corrupt value.

.Leb128Reader.java:78-84
[source,java]
----
for (int shift = 0; ; shift += 7) {
    final int group = input.readInt(true, Byte.SIZE);
    value |= ((group & 0x7FL) << shift);
    ...
}
----

Consider throwing `IOException` when `shift >= Long.SIZE`.

=== `ChannelByteInput`: busy-wait loop for non-blocking channels

The refill loop spins while `source.position() == 0`, which busy-waits if the underlying
`ReadableByteChannel` is non-blocking and returns 0.

.ChannelByteInput.java:50-54
[source,java]
----
for (source.clear(); source.position() == 0; ) {
    if (ChannelByteInput.this.source.read(source) == -1) {
        throw new EOFException("reached to an end");
    }
}
----
